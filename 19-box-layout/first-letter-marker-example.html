SOLID

  • Single-responsibility Principle: A class should have one and only one reason to change, meaning that a class should have only one job.
  • Open-closed Principle: Objects or entities should be open for extension but closed for modification.
  • Liskov Substitution Principle: objects of a superclass will be replaceable with objects of its subclasses without breaking the application.
  • Interface Segregation Principle (ISP): a class should not be forced to implement or depend on methods it doesn't need.
    A practical example of non-complying code in TS 
                
interface Worker {
    work(): void;
    eat(): void;
    sleep(): void;
}

class HumanWorker implements Worker {
    work(): void {
        console.log("Human is working...");
    }

    eat(): void {
        console.log("Human is eating...");
    }

    sleep(): void {
        console.log("Human is sleeping...");
    }
}

class RobotWorker implements Worker {
    work(): void {
        console.log("Robot is working...");
    }

    // The RobotWorker is forced to implement these methods
    // even though it doesn't need or use them.
    eat(): void {
        throw new Error("Robot does not eat!");
    }

    sleep(): void {
        throw new Error("Robot does not sleep!");
    }
}
    • The RobotWorker class is forced to implement the eat() and sleep() methods, even though a robot doesn't eat or sleep. These methods are irrelevant to the robot, but it has to implement them because they are part of the Worker interface.
    • This situation can lead to confusing, error-prone code where the RobotWorker either implements these methods with empty bodies, throws errors, or does something meaningless, like logging a message that it can't perform the action.
